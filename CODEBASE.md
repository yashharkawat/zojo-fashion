# Zojo Fashion — Codebase Reference

> This document is the single source of truth for AI agents working on this repo. Read it before touching anything.

---

## 1. What Is This

**ZOJO Fashion** is an India-first print-on-demand streetwear e-commerce store. Customers browse anime/streetwear themed clothing, pick variants (size/color), pay via Razorpay, and receive email/SMS updates through the order lifecycle. It is a production site deployed at `https://zojo-fashion.yashharkawat.com`.

## 4. Database (Prisma Schema)

File: `prisma/schema.prisma`

The Prisma client is generated into `apps/api/node_modules/.prisma/client` (not repo root) so the API's isolated `npm ci` on Railway works.

### All Models

| Model | Purpose |
|---|---|
| `User` | Customers and admins. Roles: `CUSTOMER`, `ADMIN`, `SUPPORT`, `SUPER_ADMIN` |
| `RefreshToken` | Hashed refresh tokens (never store raw) |
| `VerificationToken` | OTP, email-verify, password-reset tokens (SHA-256 hash only) |
| `Address` | India-specific: line1/2, city, state, pincode (6-digit), E.164 phone |
| `Product` | Catalog item with `avgRating`, `reviewCount`, `soldCount`, soft-delete via `deletedAt` |
| `ProductVariant` | SKU, size, color, `colorHex`, price in paise, stock |
| `ProductImage` | Cloudinary URLs; `variantColor` for color-specific images |
| `SizeChart` / `SizeChartRow` | Shared per category/type, linked from Product |
| `Cart` / `CartItem` | Persisted server cart for logged-in users; snapshots price/title/image |
| `SiteSettings` | Singleton row (`id="default"`): `instagramUrl`, `youtubeUrl`, `whatsappNumber` |
| `HomepageBanner` | CMS banners with schedule, position (HERO / STRIP / SECONDARY) |
| `Order` | Full order with snapshot of shipping address, coupon, GST |
| `OrderItem` | Snapshot of product title, variant label, SKU, prices at time of order |
| `Shipment` | 1:1 with Order. Has `trackingUrl` (Quikink/Delhivery link, kept server-side) |
| `Payment` | Razorpay IDs, amounts, raw payloads for audit. Refund events as JSON array |
| `Review` | Product reviews — `@@unique([productId, userId])`, recalculates `avgRating` + `reviewCount` on write |
| `BlogPost` | Markdown content, tags, isPublished/publishedAt draft-publish flow |

### Key Conventions

- **Money**: always stored as `Int` in paise (₹1 = 100). Never Float/Decimal.
- **Soft delete**: `Product` and `ProductVariant` have `deletedAt DateTime?` — filter with `deletedAt: null`.
- **Timestamps**: all UTC. Display layer converts to IST.
- **Phone numbers**: E.164 format (`+91XXXXXXXXXX`) everywhere.
- **Order number format**: `ZJ-2026-000123` — generated by `apps/api/src/utils/orderNumber.ts`.

### Running Migrations

Development (creates + applies new migration):
```bash
cd zojo-fashion
DATABASE_URL="..." npx prisma migrate dev --name "your_migration_name"
```

Production / CI (applies pending only, no creation):
```bash
npx prisma migrate deploy
```

The Neon DATABASE_URL is in `apps/api/.env`. The same Neon DB is used for both local dev and production (single branch). When you run `prisma migrate dev` locally with the Neon URL, changes apply to production immediately.

### Auth Flow

1. Client gets an **access token** (JWT, short-lived ~15 min) via login/register/google.
2. A **refresh token** is set as an HTTP-only cookie (long-lived).
3. The `api()` fetch wrapper in `apps/web/src/lib/api.ts` auto-detects 401s and calls `/auth/refresh` (exactly once for all concurrent requests — thundering herd prevented by `inflightRefresh` promise).
4. On the API side, `authMiddleware` verifies the Bearer token and populates `req.auth: { userId, role, email }`.
5. RBAC: `requireAdmin` / `requireFullAdmin` middleware in `src/middleware/rbac.ts`.

### Notifications (`src/lib/notifications.ts`)

Four exported composite functions, all use the same `OrderNotificationContext`:

```typescript
interface OrderNotificationContext {
  orderId: string; orderNumber: string; customerName: string;
  customerEmail: string; customerPhone: string | null;
  totalPaise: number; trackingUrl?: string; courier?: string; awbNumber?: string;
  items: { productTitle: string; variantLabel: string; quantity: number }[];
}
```

| Function | Triggers | Channels |
|---|---|---|
| `notifyOrderConfirmed` | Payment captured | Email + SMS + merchant WhatsApp/Telegram |
| `notifyOrderShipped` | Admin marks SHIPPED | Email + SMS (sends branded `/track/ORDER_NUMBER` URL, not Quikink) |
| `notifyOrderDelivered` | Admin marks DELIVERED | Email |
| `notifyOrderCancelled` | Order cancelled | Email |

Merchant alerts support three providers (tried in order): Telegram bot, CallMeBot WhatsApp, Twilio WhatsApp. Configure via env vars.

### Coupon Codes (`src/utils/orderCoupon.ts`)

Hardcoded switch statement — no DB table:

| Code | Effect |
|---|---|
| `OTAKU10` | 10% off, max ₹200 |
| `NARUTO200` | ₹200 off, min order ₹999 |
| `FREESHIP` | Free shipping |
| `WELCOME100` | ₹100 off |

To add a new coupon, add a case here AND update `apps/web/src/app/cart/page.tsx` (`previewCoupon`).

### Error Classes (`src/lib/errors.ts`)

- `ValidationError` → 422
- `UnauthorizedError` → 401
- `ForbiddenError` → 403
- `NotFoundError` → 404
- `ConflictError` → 409
- `UpstreamError` → 503

### Response Shape

All API responses follow:
```json
{ "data": <T> | null, "error": null | { "code": "...", "message": "..." }, "meta": { "requestId": "..." } }
```

Use `ok(res, data)` from `src/lib/response.ts` for success responses.

## 6. Web Frontend (`apps/web`)

### Key Architecture Decisions

- **App Router** (Next.js 14) — pages are RSC by default; use `'use client'` only when needed.
- **Server Components fetch data** — product lists, product detail, blog, settings use async RSC fetches with `next: { revalidate: N }`. No client-side initial fetch for above-the-fold content.
- **Client Components** — cart, auth state, reviews, checkout, interactive UI.
- **Redux** manages: `auth` (access token + user), `cart` (items, optimistic updates), `ui` (cart drawer open/close, modal).
- **TanStack Query** manages: admin data, orders list, order detail.
- **`api()` / `apiWithMeta()`** in `src/lib/api.ts` — the ONLY way to make authenticated API calls from the client. Handles token refresh automatically.

### State Management

**Redux slices** (`src/store/slices/`):

- `authSlice` — `{ accessToken, user, status: idle|authenticating|authenticated|unauthenticated }`. Access token stored in `localStorage` via `authStorage.ts`. Never store sensitive data here.

## 7. Order Lifecycle

```
PENDING → CONFIRMED → PRINTING → SHIPPED → DELIVERED
                                         ↘ CANCELLED
                                         ↘ REFUNDED
```

1. `POST /orders` — creates order in `PENDING`, creates Razorpay order.
2. Client pays via Razorpay checkout.
3. `POST /payments/verify` — verifies Razorpay signature, sets order to `CONFIRMED`, sends `notifyOrderConfirmed` (email + SMS + merchant WhatsApp).
4. Admin marks `PRINTING` via dashboard.
5. Admin marks `SHIPPED` + enters courier/AWB/tracking URL → `notifyOrderShipped` fires (email + SMS with `zojo-fashion.yashharkawat.com/track/ORDER_NUMBER`).
6. Admin marks `DELIVERED` → `notifyOrderDelivered` fires.

### Tracking Flow (hides Quikink from customers)

- Admin enters Quikink URL in admin dashboard when shipping an order.
- It's stored in `Shipment.trackingUrl` (DB-only; never sent directly to client HTML).
- Customer receives a branded tracking link in the shipping email (`/track/ORDER_NUMBER`).
- The customer order detail page (`/orders/[id]`) embeds the courier page in an **iframe** at the bottom. If the courier site blocks iframe embedding (X-Frame-Options), a fallback link appears.
- Server-side redirect: `GET /api/v1/track/ORDER_NUMBER/redirect` → 302 to the raw Quikink URL (used internally and for the branded email).
- Admin can resend the shipping email at any time via "↺ Resend tracking email" button on the admin order detail page (`POST /admin/orders/:id/resend-shipping-email`). This also runs an SMTP connection check and surfaces errors to the UI.

## 9. Important Patterns & Gotchas

### Adding a new API module

1. Create `apps/api/src/modules/<name>/{name}.schema.ts`, `{name}.controller.ts`, `{name}.routes.ts`.
2. Export the router from `routes.ts`.
3. Import and mount in `apps/api/src/app.ts`.
4. Use `asyncHandler()` wrapper on all route handlers.
5. Validate with `validate({ body: ..., params: ..., query: ... })` middleware using Zod schemas.
6. Access authenticated user via `req.auth.userId` (set by `authMiddleware`).

### Adding a new page (frontend)

- RSC by default — fetch data directly in the component with `await fetch(...)`.
- Add `'use client'` only if you need hooks, event handlers, or browser APIs.
- Wrap with `<PageTransition>` for consistent fade-in animation.
- For static info pages, use `<StaticInfoPage title="...">` component.

### Money

Always work in paise (integer). Use `inr(paise)` from `src/lib/format.ts` to display. Never pass floats to the API.

### Cart rules (critical)

- Max quantity per line item: 10.
- Server cart is source of truth for logged-in users.
- Guest cart lives in Redux (in-memory + not persisted across page loads).
- On login: server cart wins if non-empty; local cart pushes to server if server is empty.
- `mergeCart` (additive) is only for guest→server merge on first login with empty server. Do NOT call it on page reload — it doubles quantities.

### Reviews

- One review per user per product (`@@unique([productId, userId])`).
- User must have a paid order (`CONFIRMED|PRINTING|SHIPPED|DELIVERED`) containing the product to leave a review.
- `avgRating` and `reviewCount` on `Product` are recalculated atomically in a Prisma `$transaction` on every review create/delete.
- The inline review form on `/orders/[id]` uses the `api()` utility (not raw `fetch`) so the Bearer token is automatically included — do not use raw `fetch` here, it will get a 401.

### Order Detail Timeline

The status timeline on `/orders/[id]` uses **absolute-positioned CSS tracks** (not flex-item line segments):
- A full-width grey background bar is drawn behind all dots.
- An accent-coloured fill bar grows from 0% (Confirmed) to 100% (Delivered) based on `activeIdx / (steps - 1) * 100`.
- Dots are rendered in a `justify-between` flex row overlaid on top.
- Previous approach (line inside each `<li>` with `-mt-4`) left a visual gap before the last dot — do not revert to that pattern.

### SiteSettings

Singleton row with `id = 'default'`. Fields: `instagramUrl`, `youtubeUrl`, `whatsappNumber`. Fetched by the web app at build/request time via `fetchSiteSettings()` in `src/lib/server-settings.ts` (5-min revalidate). Also fetched server-side in `layout.tsx` to pass `whatsappNumber` to the floating WhatsApp button.

### Email (SMTP)

Uses **explicit Gmail SMTP** settings (not `service: 'gmail'`):
- Host: `smtp.gmail.com`, port `587`, `secure: false` (STARTTLS)
- Required env vars: `GMAIL_USER` + `GMAIL_APP_PASSWORD`
- This config is more reliable on cloud hosts like Railway where port 465 (SSL) is often blocked.
- `resendShippingNotification` in `admin.service.ts` runs `transporter.verify()` before sending to surface SMTP errors immediately to the admin UI.

### Admin — Mobile Layout

The admin section is fully mobile-optimised:
- On small screens, `AdminSidebar` is hidden; a `MobileTabBar` (fixed bottom nav with 4 tabs: Overview, Orders, Products, Settings) replaces it.
- `/admin/orders` renders `OrderCard` components (mobile-friendly cards) instead of a `DataTable` on small screens.
- `/admin/orders/[id]` has a sticky bottom action bar so admins can change order status with one thumb.
- Admin order list and detail auto-refresh after status changes via React Query invalidation.
- The admin layout adds `pb-24` to the content area to clear the bottom tab bar.

### Google Analytics

GA4 script loaded in `layout.tsx` via `next/script` with `afterInteractive`. Page view tracking on route change handled by `GoogleAnalytics` component (uses `usePathname` + `useSearchParams` inside a `<Suspense>` boundary to avoid prerendering errors).

### GoatCounter (Visitor Count)

- Tracking script injected in `layout.tsx` via `next/script` (`data-goatcounter="${GC_URL}/count"`).
- `VisitorCount` component in the footer fetches `/api/visitors` (Next.js route handler) which proxies `${GC_URL}/counter/%2F.json`.
- The proxy caches for 60 seconds (`next: { revalidate: 60 }`).
- **Requires "Make statistics public"** to be enabled in the GoatCounter dashboard — otherwise the JSON endpoint returns 403 and no count is shown.
- Count only appears when it's non-zero (component returns `null` otherwise).
